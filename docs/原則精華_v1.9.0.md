## A. 資料紀律（Data Discipline）—— 主防敵人 ① 假資料　〔#1-7, #17, #18〕

### 1. 零幻像 / Source-Pure ★（系統的命）
- **WHAT**：任一特徵值必須是「真實 FinMind/FRED API 值經數學轉換」而得；imputed / zero-fill / hardcoded / 推估 = 幻像。**算不出真實值即不寫入（缺列），不存 fake / zero-fill**（扣靈魂「不存沒有 API 來源的值」）。
- **WHY**：用假資料跑模型沒有意義。
- **ENFORCE**：feature 層算不出 → 缺列；universe 層 **completeness gate** 檢查**全部同版面板**，任一面板任一特徵缺值之股即排除核心；模型只吃完整核心股（無 imputed 可濾）。

### 2. API 即權威
- **WHAT**：表名 / 欄位 / 型別 / 大小寫一律以 API 回應為準（逐字）；下游 schema 查 DB `information_schema`，不另立手維白名單。
- **WHY**：手維 registry 會漂移；API 是唯一事實。
- **ENFORCE**：DB↔API byte-level 對帳；schema helper 由 DB 推導。

### 3. 純通用 Ingestion
- **WHAT**：無 dataset 白名單；通用 auto-schema 自動建任意 API 表。全 dataset 動態列舉（`sync.daily_datasets()`）、無 hardcoded 清單。
- **WHY**：不該存在「只認 N 張表」的 sync 引擎。
- **ENFORCE**：任意 dataset 名傳入 ingester 即 auto-schema 建表（如 `daily_maintenance.py --datasets X`）；**資料本質排除＝#4 日為最小單位（intraday 守門）**——不以抓取維度/方式或「非個股標的」排除（總經/衍生品/外股/商品/匯率/可轉債皆想辦法抓）。**`ingest.OUT_OF_UNIT`＝「資料量規模物理界限」之排除〔唯一權威家＝本條；#4/#18/憲章皆引此、不複述〕**（日級合格 #4、但全市場全史資料量物理裝不下/抓不完，非實作懶、非抓法不會）：**券商分點 / 權證 / 鉅額分點**——probe 實證（2026-06-23/24）3 表 endpoint 行為一致（per-(股,日) 逐日結構）+ schema PK bug 風險（單列 sample 觸發 `detect_keys` 提早 return），全市場全史**資料量物理不可行**＝分點 per-(券商,股,日) 數十億列 TB 級（DB 裝不下）、權證數千萬列＋~178萬 calls、鉅額分點 endpoint 不吃 end_date+單 call 同股同日 17+ 列券商×trade_type schema PK bug 16+→1 覆蓋 →**3 表確認排除**（非「待有能力」暫緩；是資料量+schema 雙重物理界限）。

### 4. 日為最小單位（唯一資料本質排除準則）
- **WHAT**：系統時間最小單位＝「日」；不收 tick / 分K / 5 秒，intraday 來源只存日級衍生值（每日一列）。**「<日」之 intraday 是系統唯一的資料本質排除準則**——凡 API 有的**日級**真實資料皆應抓入（總經 / 衍生品(期/權) / 外股 / 商品 / 匯率 / 可轉債 / 籌碼，皆為有價值之情境或特徵輸入），**不以抓取維度/方式（需特殊 id）或「非個股標的」為排除理由**。
- **WHY**：資料價值決定抓不抓，抓取實作應配合（想辦法抓）；唯「<日」之 intraday 違反系統時間本質。用單一抓取條件（引擎只會 market/per-stock/by-date）去砍真實資料，是讓實作限制反噬資料完整度。
- **ENFORCE**：intraday dataset 守門拒絕入庫（日級守門＝資料本質 gate）；需特殊維度 id 之 dataset → 補通用多維度抓取（#18），非排除；**資料量規模物理界限者 → `OUT_OF_UNIT` 排除，判準全文見 #3**。〔辨明：intraday＝唯一「資料本質」排除；`OUT_OF_UNIT`＝「資料量物理界限」排除，二者不同類〕

### 5. 型別紀律
- **WHAT**：字串最小 `VARCHAR(255)`、超過 255 改 `TEXT`（無中間寬度）；數字最小 `NUMERIC(20,6)`、超出自動擴大 precision/scale。
- **WHY**：一致、可預期、不截斷不溢位。
- **ENFORCE**：generic_schema 推導 + 既有表 auto-widen（只擴不縮）；**型別不相容值亦屬「擴」**——後續批出現與既有欄型不容之值（數字欄遇契約月 `200710/200711`、日期欄遇 sentinel `-1`）→ 降級為字串容納 API 全值域（守 #1 不丟列、#2 API 即事實）。

### 6. 冪等 + 斷點續傳
- **WHAT**：同範圍重跑安全；中斷可從 DB 最後狀態續抓。
- **WHY**：長跑 sync 必須可恢復、不重複、不掉資料。
- **ENFORCE**：`ON CONFLICT DO UPDATE` + DB-driven resume + 主鍵首建固定（不被小樣本推窄）。

### 7. DB↔API 對帳 Attestation
- **WHAT**：定案資料須 byte-level 等同 API。對帳分 4 類——matched / `value_mismatch`（DB≠API）/ `missing_in_db`（API 有 DB 無）/ `extra_in_db`（DB 有 API 無）；attestation 通過 = `value_mismatch=0 ∧ extra_in_db=0`。
- **WHY**：DB 無幻像之可驗證鐵證；API 即權威（#2），DB 應與之一致。
- **ENFORCE（偵測 + 處置）**：reconcile audit 偵測差異後——
  - `value_mismatch` / `missing_in_db`（定案資料）→ **用同一條 sync 路徑重抓該 (股, dataset, 區間)**（強制 re-fetch、繞過 #6 resume 的「已同步即跳過」）→ 冪等 `ON CONFLICT DO UPDATE` 自然覆蓋為當前 API 值。**無獨立修正程式：correction＝重跑正常 sync**（同一 upsert 路徑，守 #1 no-hand-patch、#2 API 權威、#6 冪等）。
  - `extra_in_db`（DB 有 API 無）→ **疑似幻像 / PK 碰撞之紅旗 → 調查根因，不自動補也不自動刪**（亦可能為已下市、API 不再回傳之合法歷史）。
  - **未定案 / 非穩定期之差異 → 可接受**（非紅旗、非 sync bug）：當日仍結算中（次日刷新）、季頻財報最新季發布／重編中（法定申報期後定案）、還原價 `PriceAdj` 隨未來除權息回溯重算致歷史列改寫（對帳以抓取時點 API 當前值為準，byte 定案性僅於無後續除權息區間成立）。
  - 全部差異**記錄留存**（哪筆 / 舊→新值 / 偵測時點）供 audit，區分 restatement（合法）vs sync bug（須修程式）。

### 17. API 速率公民 / 主動限速（不過載服務端）
- **WHAT**：對 FinMind / FRED（及任何 API）服務端**不得短時請求量過高**；抓取一律**主動限速**——每請求最小間隔 ＋ 整體壓在服務端速率線下、平滑 burst；遇限速/封鎖（402 / 429 / 403）依回應 `retry_after` 退避等候再續，不硬衝。
- **WHY**：短時 burst 會觸發**服務端 IP 封鎖**（augur 曾實證 burst → `403 ip banned`），封鎖期間**全系統無法取得任何真實資料**→ 比慢更糟。做好 API 公民，才有穩定的真兆來源。
- **ENFORCE（三層防護:步調→閘→冷卻,皆「撞前主動」非「失敗才退避」）**：fetch 層**內建**——(1) `_pace()` 每請求最小間隔（基礎步調）；(2) **主動額度閘**：閉環輪詢服務端**權威額度錶**（`/user_info`），額度近滿**撞前先停**、退夠自動續（實證：額度為 rolling 視窗、計數含未知成分 → **問錶、不本地推算**）；(3) 403 **長冷卻**保險網：不短退避反覆撞（防**重試風暴**把額度限流惡化成 sustained 封鎖）；honor `retry_after`；long-running sync 不得以高併發 / 無間隔狂打服務端；**驗證與全史走同一 `fetch`、同一防護**（驗證怎麼溫和，全史就怎麼溫和）。
  - **operational 最小間隔（`MIN_INTERVAL`）為透明揭露之 operational 參數**（非 feature 值，不違 #9）、住於 `finmind.py` 單一處（#12 SSOT）、**可依實證逐級調整（#19）、不寫死於本原則**；當前值與 re-ban 實證光譜見 `finmind.py` 與憲章修訂歷程。

### 18. 抓取依 API 探測決定方式與範圍 / 不空抓（API-Driven Fetch, No Empty-Scan）
- **WHAT**：要抓**每張表前**，先**透過 API 探測**決定 (a) **抓取方式**（market 整批 / per-stock 逐股 / by-date 逐日全市場）與 (b) **資料起始日**；只抓 API **實際有資料**之範圍，**不對已知無資料之期間發請求（不空抓）**；不寫死起點（如 1990）逐日空掃無資料年份。
- **WHY**：抓取方式與資料範圍是 **API 的事實**，應問 API、不假設。寫死起點逐日掃無資料年份 = 空抓 → 浪費請求/時間、不友善服務端（違 #17 精神）、且非「依 API 實際資料」。
- **ENFORCE**（逐點）：
  - **方式**＝adaptive 探測階層：market-probe → per-stock canonical-probe → by-date → **by 維度 id**。
  - **維度 id 全集**：**需特殊維度 id（`currency` / `maturity` / `index` / `futures_id` / `option_id`…）不是排除理由 → 補通用多維度抓取**；id 全集**依序由 (a) FinMind `/datalist` API（總經/契約類、動態最完整）(b) 官方文檔/IndexCodes + live-probe 證實之文檔種子（無 datalist 之少數，如報酬指數 `TAIEX`/`TPEx`）(c) roster/Info 表（per-stock）取得**——**程式一律先問 `/datalist`、不抄靜態文檔**（實證 `/datalist` > 靜態文檔）；權威來源 + probe 證實**全集**，**無 hardcoded 臆測白名單**。
  - **起點**＝resume 從 DB `max(date)`，首次最小 API 探測（自近往遠找邊界、多日取樣避假日、不探未來日），**禁 `full_start` 逐日空掃**。〔**辨明**：資料最早日由 **API 回傳/探測決定**、系統不預設；`FULL_START`（如 1990）僅單次 call 之**查詢下界參數**——**非全史保險**：API 以 `start_date` 為下界截斷、不回更早資料（GoldPrice 真起點 1979 被 1990 截斷之 2026-06-16 教訓），earliest 卡 `FULL_START` 者由 catalog `_refine_earliest` 往更早探真起點——**非最早日、亦非逐日掃起點**；「不寫死起點」禁的是 by-date 逐日空掃、非此下界參數〕
  - **單日型 / 事件型**：單日型 dataset（FinMind size-too-large、強制一次一天、`end_date` 須 none，如 `TaiwanStockNews`）→ `fetch` 偵測訊息自動移除 `end_date` 逐日抓；其 `date` 為 datetime 事件型者 **by-date 不跳週末**（週末取捨依資料事實、不假設無資料，守資料來源一致性）。
  - **資料量物理界限**：唯資料量規模物理不可行者（分點/權證/鉅額分點）→ `OUT_OF_UNIT` 排除，**判準全文見 #3**。

### 資料完整性判準（Completeness Criterion · as-of）〔承 #6 as-of · #7 對帳 · #18 範圍〕
- **WHAT**：**資料完整 ＝ 每支個股及其相關資料，自各自 API 起始最早日（#18 探測決定）起、至 as-of 截止日 `2026-05-31` 止，所需資料全數滿足無缺**。系統級「完整」（如 84 表）＝每張 in-scope 表之每股（per-stock）/ 每日（by-date）/ 每維度（by-dim-id）覆蓋至 as-of 日皆滿足。
- **WHY**：市場每日產生新交易資料 → 若以「到今日」為完整目標，判準逐日浮動、永遠追不完、對帳目標不定。**固定 as-of 截止日**使完整性成為**可定案、可驗證、可稽核**之判準（承 #6 as-of functional、#7 byte 定案性、#15 誠實）。
- **ENFORCE**：sync 目標 / 對帳範圍 / 「完整」宣稱一律以 as-of `2026-05-31` 為界；**最早日**由 API 探測決定（#18）、**截止日**固定為 as-of；as-of 後新資料屬增量維運（daily_maintenance）、不納入「截至 as-of 已完整」之判定。as-of 值為**治權參數**——隨專案推進可更新、須**用戶決策後入憲**（不由 AI 擅改）。
- **FREEZE→解凍〔v1.9.0 / 2026-07-12 用戶親核拍板〕**：as-of `2026-05-31` 之全系統資料期限凍結（v1.8.0 FREEZE，2026-07-09～2026-07-12）**已解除**——自解凍拍板起轉 **live 增量維運**：每日增量 sync 與 panel 續建為常態、新 as-of'＝**滾動（每日 sync 至最新交易日）**；live 准入依 `prediction_unfreeze_gate`（`unfreeze_06dcb178267d`，U1-U6 判準已凍結）之 evaluate 紀律——**GATE evaluate pass 前，live 數字不得入任何「確立級」宣稱**（憲章④硬綁不變）。歷史完整性判準（至 2026-05-31）之定案性不受影響、凍結期快照留檔可複現。as-of 仍為治權參數，更新須用戶決策入憲（AI 不擅改）。

## B. 建模紀律（Modeling Discipline）—— 建模階段橫跨三敵人（#8②、#9①、#10①③、#11 #12③）　〔#8-12〕

### 8. Anti-Leakage
- **WHAT**：只用 as-of ≤t 已知資訊、t+1 套用；walk-forward IC **必附 purged 口徑**（剔除標籤窗與 test 重疊之訓練面板）。
- **WHY**：用了決策當下尚未實現的 label = 樂觀偏誤 = 假成績。
- **ENFORCE**：發布日 gate + embargo purged-CV；**含 survivorship / 選股偏差防護**（歷史宇宙須含當期已下市股，不只「活到今天」的；選股本身不用未來資訊）；單報 raw IC 視為違規。

### 9. 思想 ≠ 特定值
- **WHAT**：可受 Pareto / 康波等思想啟發；但**不把特定值硬編進特徵**——八二法則 `0.80 / 0.20`（含 20/60/20、10% decile 等衍生切點）、康波 `40-60 年`、theme keyword 分數，皆禁入 feature 公式。
- **WHY**：特定預測值 = AI 幻像；思想 ≠ 預測。
- **ENFORCE**：feature 公式禁 hardcoded 知識字典 / 閾值；operational 層（風控配比）之特定值須透明揭露且不入 feature。

### 10. 核心股：質 > 量
- **WHAT**：核心股以資料正確（source-pure）+ 精準（合理性，排離群/錯誤值）為最高判準；**可以變少**。
- **WHY**：要的不是核心股多，是資料對且精準。
- **ENFORCE**：pan-historical 完整度 gate（核心 ＝ 全部 source-pure 完整股，**不評分、不排名上限**）；不以核心股數量為目標。

### 11. 五鏡特徵治理
- **WHAT**：判斷特徵存廢**不得單一指標**（尤不得單看 gain）；五鏡合看——有號 IC+sign 穩定 / 共線群 / leave-one-out 必要性 / ensemble SHAP / purged-CV。
- **WHY**：單一指標會把「共線冗餘」誤判為「真噪音」，或留下模型已忽略的死重。
- **ENFORCE**：五 audit harness 合判；「不顯影（SHAP≈0）且 ablation-safe」必移。

### 12. 單一引用源（SSOT）
- **WHAT**：「用哪些 panel」+「metric 怎麼算」由**單一 helper** 決定；禁各 validator 寫死或 inline。
- **WHY**：跨模型比較必須同基準，否則不可比、不可信。
- **ENFORCE**：全 validator import 共用 panel-date + metric helper。

## C. 風險與治理紀律（Risk & Governance）—— 防敵人 ③ 自我欺騙　〔#13-15, #19〕

### 13. 空頭防護：規則地板優先
- **WHAT**：空頭防護**先建規則型風控地板**（波動目標 × 趨勢，不需預測）；regime 預測只當可選增強層。
- **WHY**：實證——規則地板獨力即大幅降回撤；預測當主角反而較差。
- **ENFORCE**：分層歸因（floor vs floor×regime 增量）；增強層須證 ΔMaxDD<0 ∧ ΔCalmar>0 否則標冗餘。

### 14. 經濟價值判定
- **WHAT**：overlay / 擇時 / 操作週期以 MaxDD / Calmar / Sharpe + 逐空頭子期判定，**非 AUC / 準確率**。
- **WHY**：市場擇時極難；準確率高不等於賺錢或保本。
- **ENFORCE**：walk-forward 經濟價值檢驗；最佳操作週期以「淨報酬 × 信任度 × 穩健度 × 風險」綜合，不裸追最高淨報酬。

### 15. 誠實回報 / 無 Aspirational Inscription
- **WHAT**：每個量化數字可 trace 回 (a) 程式輸出 / (b) DB query / (c) API 回應；**不把「計畫 / 未跑完」當成果**寫入文件；stochastic 指標 ≥3 次取統計；**可重現**（固定 `random_state`、pipeline 確定性 → 半年後重跑結果一致，即靈魂的成功定義）。
- **WHY**：文件失真比沒文件更危險（會驅動錯誤決策）。
- **ENFORCE**：inscription 前 source 檢查清單；結果未跑完不入（待 source-traceable 補入）。

### 19. 可控風險下逐級逼近最佳奇異點 / 試錯即進步（Calculated-Risk Optimization）
- **WHAT**：augur **解決問題的預設做法**——在**可控、可恢復、可即時退場**的風險邊界內，朝有利方向**逐級逼近、主動試錯**，找出**最佳奇異點**（效益最大、恰落在崩潰邊界之內的操作點）。具體：某參數當前級經驗證可行 → 再往進一級試（如限速 `0.6s` 驗證可行 → 試 `0.5s` → 以此類推），直到某級**持續性失敗**，退回**上一已驗證安全級**＝該問題的最佳奇異點。**不被單次過往觀測永久錨定**（一次失敗只是「某情境的一個觀測」、非永久天花板），於當前條件**重覆實證、重覆驗證再下定論**。**不怕一時的（可恢復）錯誤——所有進步都來自試錯**。此非僅限速率，而是**對所有問題的通用方法論**。
- **WHY**：一味保守會把「一次情境性失敗」當永久天花板、長期白白犧牲報酬；無紀律的冒進則毀掉資源。真正的最佳＝**可控風險內逼近極限的那個奇異點**，它只能靠**逐級試錯 + 重覆驗證**找到，不能靠臆測或畏懼——**怕犯錯就不會進步**。
- **ENFORCE（可控風險三要件，缺一即非「可控」、不得試；＋兩條凌駕邊界）**：
  - **(a) 可恢復**：後果能自癒、無永久損害、無資料損失（如 API ban → cooldown 自癒 + resume-safe；**非**毀資料 / 永久封號 / 不可逆）。**只試「可恢復的錯誤」，不可恢復的不試**。
  - **(b) 有界**：最壞 = 退回**上一已驗證安全級**；損失上界已知（如數次 cooldown 時間）。
  - **(c) 可即時偵測 + 即時退場**：明確即時訊號（速率崩塌 / 錯誤碼 / 指標異常）＋ 即時退一級機制；**一觸即退、不硬衝**（承 #17 honor `retry_after`）。
  - **逐級 + 重覆驗證再定論**：每級須**持續性實證**（短測會騙人、須在真正壓力條件下持續觀察）方算「該級可行」才往下一級；新安全值 source-traceable 才入憲；**單次成功/失敗皆不定論**。
  - **凌駕邊界一（範圍）**：本原則**只作用於 operational / 方法層**（速率 / 並發 / 參數 / 解法選擇 / 資源利用 / 排程）；不作用於資料層。
  - **凌駕邊界二（最重要）**：**資料誠實零容忍、不是試錯對象**——三大敵人（假資料 / 偷看未來 / 自我欺騙）**不得拿來「試錯」**；不得為「報酬」捏造資料、跳過驗證、樂觀回報。**試錯試的是「方法與參數」，不是「資料真假」**；「報酬」＝真實吞吐 / 覆蓋 / 預測力，非灌水數字。

## D. 開發 / 協作紀律（Development & Collaboration）—— 守重建完整性 ＋ AI 自驅動實證（防敵人 ①③ 經 code 回流 / 憑「以為」闖禍）　〔#16, #20〕

### 16. Clean-Room 重建 / 零 stock_backend 參考
- **WHAT**：augur 的**程式（code）產生一律 clean-room**——只依 augur 自身 5 治權檔（靈魂 / 原則精華 / 憲章 / CLAUDE.md / README）＋ augur 自身 schema 目錄 ＋ live API 實證 建立；**產生任何 code 時，不讀、不參考、不移植 stock_backend 之任何 code / 資料 / 報告 / 數字 / 設定**。
- **WHY**：augur 是 stock_backend 的紀律化重建。參考其 code 會把舊系統的硬編值（敵人①）、未驗證假設、255-treaty 累積病、aspirational 數字（敵人③）原樣回流——重建即失去意義。乾淨底座只能從乾淨源頭長出。
- **ENFORCE**：唯一 sanctioned 的 stock_backend 觸點＝憲章**附錄 B 考古索引**（查某 doctrine 之**演化史**，文件層）＋已抽象進 augur 自身文件之**思想啟發**（#9 思想≠特定值，概念層，不含 code / 特定值）——二者**皆不得回流 code**。任何 augur 程式出現 stock_backend 之 code 片段 / 路徑 / 變數名 / 數字 / 移植痕跡 ＝ 治權違規。

### 20. 自驅動 × 實證決策（AI 開發 augur 的協作鐵律）
- **WHAT**：經用戶授權，AI 開發 augur 採「**自驅動 × 實證**」一體兩面——**(主動)** 不待逐動指令、**自己 prompt 自己**（loop）持續朝靈魂目標推進**護欄內**工作；執行層「如何最佳達成目標」由 AI 主導（技術路徑 AI 更清楚）；發現問題／錯誤／不一致 → 自驅修正、試各種解法。**(實證)** 凡判斷／結論／做法／operational 決策，**一律先以實證（probe／實測／API／code／DB query）驗證再下，嚴禁憑「我以為／推論／記憶」決策**。主動給速度、實證給正確——**無實證的主動必闖禍**。
- **WHY**：「人說一動、AI 做一動」既浪費 AI 能力、又拖慢逼近靈魂目標；但無實證的主動會憑想當然闖禍（augur 曾實證：憑臆測的 pattern 誤殺進程）。要**快又要對**，唯有主動與實證並行。對映三敵人③：憑「我以為」下決策＝自我欺騙的一種——本條即 **#15 誠實的擴展**（誠實不只及於數字，及於一切判斷與做法）。
- **ENFORCE（決策層人拍板 / 執行層 AI 自駕 ＋ 實證鐵律 ＋ 護欄）**：
  - **決策層人拍板**：方向 / 判準變更 / 資料真假 / 外部副作用 / commit-push / 扣扳機（下單）→ AI 停下問。
  - **執行層 AI 自駕**：方法 / 路徑 / 參數 / 自我糾錯 / 試錯（承 #19）→ 授權後主動、不逐步等指令；做完呈過目。
  - **實證鐵律及於一切**：不只量化數字（#15），連 operational 判斷（殺哪個進程、走哪條路徑、調哪個參數）也須**先驗證再做**；不確定 → 先 probe / 查清，不憑記憶或推測下手；**主動自問「有沒有更近的事實、更正確的做法」**，對自己的答案存疑、以事實檢驗（防「以為懂其實自欺」——主動 ≠ 自信犯錯）。
  - **執行層天職＝確保完整性**〔2026-07-04 自憲章第四部索引欄移入，內容不變〕：程式/資料/metadata **全欄全表全驗證**；完整是無條件交付、更是決策層判斷「要不要用/抓」之前提；**尤禁「我以為不重要/不需要」當跳過理由**（＝漏做之源、系統出錯主因，2026-06-16 實證教訓）；嚴禁以此預判砍完整＝越權（替用戶決定）＋失職（不給判斷依據），亦不令用戶重覆提醒同一事；**窮舉/驗證須試到真邊界**（探到資料真的沒有為止）——連範圍/邊界/慣例值（如 `FULL_START`）本身都是待驗證對象，不可用「慣例值」當停損後自稱窮舉；見可疑結果（如「剛好卡在邊界」）追查到底、嚴禁自我合理化（＝自欺，2026-06-16 GoldPrice 1990 floor 截斷 1979-1990 真資料之教訓）。
  - **碰護欄即停**：API 放量（#17）／ commit-push ／ 變更治權判準（#19）／ 破壞性 ／ 真兆最終確認（#15）。
  - **工具層對映**：CLAUDE.md #26（有界自主推進）/ #27（可控風險試錯）為本條之工具／操作層落地。
