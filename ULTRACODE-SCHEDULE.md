# ULTRACODE 分層窮盡對抗檢驗排程（L0–L7）

* **性質**：[I] 作業排程與方法登錄。本檔不改任何規格一字；各輪產出為 **[I] 審查素材**，處置權專屬 Constitution Steward（§8.5／§8.6 修訂、§8.1 解釋）。
* **緣起**：八層已於 2026-07-19 全數蓋章（G5 形式封印 8/8，見 `LAYER-SEALING-SCHEDULE.md`）。惟**蓋章＝G5 五閘齊備，不等於各層本體已受系統性攻擊**——L0 至此僅受過一次三鏡首審。本排程為八層各自之**窮盡對抗檢驗**，把每一層當成攻擊標的而非校對對象。
* **與既有審查之別**：G1 三鏡首審為「找 findings」；ultracode 為「**扮演攻擊者窮盡規避路徑**」＋「**親自清點每一個被宣稱的數字與落點**」。後者專打本專案最常見之二類缺陷：**幽靈落點**（TR 宣稱承接、實際去讀那條卻沒承載）與**不實宣稱**（「缺 0 條」「已完備」）。

## 方法骨架（各層共用）

```
Find      多維獨立對抗攻擊（每層 6–8 維，維度依該層特性訂）
   ↓
Verify    每一 finding × 三鏡獨立反駁（文本體系／形式邏輯／實務規避）
          **反駁官預設 refuted=true**；≥2 鏡反駁即出局
          counter_evidence 必須為逐字原文＋行號
   ↓
Critic    完整性批評官——問「什麼還沒被檢查」
          含：質疑各維之 scope_checked 自陳、抽查推翻理由是否可疑
   ↓
Synthesize → Steward 呈核報告（含**誠實界限節**，不得省略）
```

**共用鐵律（寫入每一代理之指令）**
1. 審查官不得修改任何檔案一字元——只讀、只分析、只回報。
2. 處置建議須標**修訂門檻**；**§8 及其構成性依據（§0.2-0.3、§2）之修訂一律原則級、非 minor**（2026-07-18 越權糾正之教訓）。
3. 不得以 linter 綠燈為合憲依據（已三度實證與實質錯誤並存）。
4. **不得採信任何文件之自陳**——L2 CS.10、L3 TR.Y、L4 TR.Y、L5 首審皆曾查獲「已成就／缺 0 條」不實。
5. **幽靈落點必查**：凡查落點必須親讀被指之下層條文，判斷其是否真的承載該義務。
6. 每一 finding 必附行號與逐字引文。

**方法論已知風險（載明以備複核）**：≥2 鏡反駁即出局之門檻**可能過寬**——L2 之 CS.10 major 曾被三鏡「雙反駁出局」而事後窮舉證明為真缺陷。故各層之完整性批評官均被明文要求**抽查推翻理由並質疑之**。
