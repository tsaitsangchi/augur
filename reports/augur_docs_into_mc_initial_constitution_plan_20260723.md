**日期**：2026-07-23

**觸發**：用戶問——「可以以元憲章為基礎架構，將 docs 下有關憲章、精神、原則整合進來元憲章，成為此 augur 專案的最初版憲章嗎？」

## 0. 一句結論（給 Steward／父代理轉述）

**可以，但須先定義「最初版憲章」≠「把現行雙軌壓成單一 L0 檔」；且不建議在 2026-10-14 合規／殘留日曆前做實質上收。**

## 2. 分析答題（A–F）

### A. 可以嗎？（程序／Sole Steward／命名與位階）

| 面向 | 判定 |
|---|---|
| **程序** | Sole Steward 得依 §8.5／附則議決。§0.5 增列／改定位＝minor；把領域法律／精神**寫進 MC [N] 本文**、改 PA／五原則／§8＝原則級（major＋全下層複審 §8.6）。 |
| **「整進元憲章」字面** | 技術上可行，但是 **L0 職能膨脹**：MC 今日是跨領域不變式＋治理程序，不是台股管線操作手冊。 |
| **「成為最初版憲章」命名** | **位階上不宜**。L0 已是 v1.6、L1–L7 已生效；再稱「最初版」＝時空倒敘，易造成引用與合規聲明混亂。若要「單一領域憲章」產品名，應定義為 **Domain Constitution Package v1（受 MC 約束）**，或未來 **MC v2（major）**——二者皆非「最初」。 |
| **與現行登錄衝突** | 真上收須同步：§0.5 改表、WM.6／Annex A、合規聲明、clean-room 五檔清單、README／CLAUDE 從屬句——否則出現**三權威**（舊 docs＋新併文＋specs）。 |

**答 A**：程序上 **可以**；若目標是「最初版」品牌，須改定義為「領域憲章包首次統一封裝」或「MC major 重構」，**不可**把現況假裝成從零立憲。

### B. 應否現在做？

| 理由 | 方向 |
|---|---|
| ultracode／v1.6 **剛齊**；殘留與 L5／L7／WM 補正綁 **2026-10-14** | **暫緩**實質上收 |
| 五檔合規聲明尚未補完（同日曆） | 先完成登錄義務，再談合併 |
| 原則 #7 vs P4.E5 未拍板 | 上收會污染 L0 |
| clean-room／SSOT：雙寫過渡期極易「以為懂其實自欺」 | 違反靈魂三敵人精神 |
| 用戶動機（入口單一、精神可尋）可用 **案 D** 低成本滿足 | 比例原則：不必動 [N] |

**答 B**：**不建議現在做實質整合**；建議現在只做入口／地圖／合規聲明準備（仍屬執行層、可另開工單，**本輪仍零改 [N]**）。

### C. 整合模型選項（≥3 案）

#### 案 A — docs 精華上收為 MC 新部／附錄 [N]

- **做法**：自靈魂／原則／大憲章抽取「跨領域不變式」，寫入 MC 新章或 Normative Appendix；docs 改 [I] 領域闡釋或逐步廢止義務句。  
- **優**：單一 L0 閱讀面；與「以元憲章為基礎架構」字面最貼。  
- **劣**：極易把台股操作（as-of、FinMind 步調、12-PHASE）抬成 L0 → 破壞 §0.6 概念層獨立性與「MC＝元」定位；觸發 major＋全下層複審；母集 102 與條款編號紀律壓力大。  
- **門檻**：多半原則級或至少重大 [N] 增補。

#### 案 B — docs 維持 SSOT，MC 只索引（強化現狀）

- **做法**：維持 §0.5 登錄；MC Appendix／README 增「領域治權地圖」；義務全文仍住 docs／specs；WM.6 刪名測試繼續約束靈魂。  
- **優**：零（或極少）MC [N] 改動；對齊先例 002；風險最低。  
- **劣**：讀者仍覺「雙軌」；docs 自稱「憲法」與 META「憲章」易混淆（可用改名／導讀緩解，不必併文）。  
- **門檻**：導讀／[I]＝patch／minor；改 §0.5 措辭＝minor。

#### 案 C — 合併為單一 `AUGUR-CONSTITUTION`（或「Constitution v0／v1」）取代「元憲章＋docs 雙軌」

- **做法**：新檔吞併 META＋靈魂＋原則＋大憲章（或取代 META 檔名）。  
- **優**：產品敘事最乾淨（「一本憲法」）。  
- **劣**：**最高成本**——L1–L7 引用格式、合規聲明、MCP 工具、裁決鏈、clean-room 全改；「元」層與領域層混一後無法用 lex superior 自洽裁領域緊張；與剛完成之 v1.6／ultracode 投資相衝。  
- **門檻**：實質＝MC major＋規格全面重新認證。

#### 案 D — 分層保留（靈魂／原則／領域憲章／MC）＋統一入口文件（推薦近程）

- **做法**：法律層級不變；新增或擴充**單一入口**（建議名：`GOVERNANCE.md` 或擴 README「治權地圖」）明確四層：  
  1. L0 META（不變式＋治理）  
  2. L1–L7 specs（正式規格）  
  3. docs 三件套（領域靈魂／法律／架構）  
  4. CLAUDE／README（短半衰期）  
  並規定「義務查找順序」與「禁止雙寫」。  
- **優**：滿足「整合進來」的**認知整合**；不動 [N]；可在 10-14 前與合規聲明並行。  
- **劣**：磁碟上仍多檔——若用戶堅持「一個 md 檔」則不足。  
- **門檻**：入口檔＝文件／patch；不改 MC [N] 則零憲政成本。

#### 案 E（可選遠程）— 「領域憲章包」封裝，不上收 L0

- **做法**：定義 `AUGUR-DOMAIN-CONSTITUTION`＝（靈魂＋原則精華＋大憲章）之**版本化捆綁＋單一合規聲明封面**，仍受 MC 約束、§0.5 維持分層登錄。  
- **優**：給予「最初／第一版領域憲章」合法命名空間，不偽造 L0 史。  
- **劣**：多一層包裝；須防封面與三檔正文雙 SSOT（封面只準索引）。  

### D. 不可整併或須保持分離者
